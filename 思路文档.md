# miniAtoms 实现思路文档

## 1. 项目简介

**miniAtoms** 是一个 AI Agent 驱动的应用原型生成器。用户可以通过自然语言描述想要创建的应用，系统会调用大模型生成结构化应用方案，并将结果渲染为一个可视化、可交互的网页应用预览。

项目核心流程如下：

```txt
用户注册 / 登录
  ↓
首页输入应用需求
  ↓
AI Agent 分析需求并生成 AppSpec
  ↓
前端根据 AppSpec 渲染可交互应用预览
  ↓
用户保存项目到数据库
  ↓
用户查看历史项目与项目详情
  ↓
用户继续输入修改指令，让 AI 迭代已有项目
```

本项目的目标不是简单做一个 AI 聊天框，而是尽量还原一个轻量级的 AI App Builder 工作流：**从想法输入、智能生成、交互预览、持久化保存，到后续继续迭代**。

---

## 2. 项目目标与题目要求对应

本次笔试要求完成一个可运行的 Atoms Demo，重点关注如何借助 AI 工具，将想法快速转化为一个可运行、可体验、可扩展的产品原型，而不是单纯跑通代码。

miniAtoms 对应实现如下：

| 题目要求 | miniAtoms 对应实现 |
|---|---|
| 可运行网页应用 | 使用 Next.js 实现完整 Web App |
| 类似 Atoms 的 AI Agent 体验 | 首页输入需求，Builder 中展示 Agent 生成流程 |
| 代码 / 应用生成能力 | 大模型生成结构化 AppSpec，并渲染成应用原型 |
| 可视化网页展示 | App Preview 动态展示生成应用 |
| 真实交互 | 预览应用支持 tab 切换、表单输入、新增记录、删除记录、统计更新 |
| 数据持久化 | 使用 Neon PostgreSQL 保存用户项目 |
| 注册 / 登录流程 | 使用 Clerk 实现登录、注册、用户隔离 |
| 在线可测试链接 | 支持部署到 Vercel |
| 代码链接 | 项目代码托管到 GitHub |
| 延展能力 | 支持对已保存项目继续 AI 迭代 |

## 3. 核心功能设计

### 3.1 极简清透科技风格首页

首页采用类似 极简清透的科技产品风格设计：

- 大面积蓝紫粉橙渐变背景；
- 顶部极简导航；
- 中央主标题；
- 居中的 Prompt 输入框；
- 右上角登录 / 注册入口；
- 已登录后显示 Projects 和用户头像。

首页主标题为：

```txt
Build apps with AI Agents
```

用户可以直接在首页输入需求，例如：

```txt
帮我生成一个个人记账应用
```

如果用户未登录，系统会弹出 Clerk 登录窗口；登录成功后自动继续刚才的 Prompt，并跳转到 Builder 页面。

这个设计避免了传统 “先登录再开始” 的割裂体验，让用户可以先表达意图，再完成身份验证。

---

### 3.2 注册与登录系统

项目使用 **Clerk** 实现注册登录。

主要功能包括：

- 登录弹窗；
- 注册弹窗；
- 用户头像；
- 登出；
- 用户状态判断；
- 受保护路由；
- 服务端 API 鉴权。

受保护页面包括：

```txt
/builder
/projects
/projects/[id]
```

受保护 API 包括：

```txt
/api/generate
/api/projects
/api/projects/[id]
/api/iterate
```

每个项目数据都会绑定 Clerk 的 `userId`，保证不同用户只能访问自己的项目。

---

### 3.3 AI Builder 工作台

用户登录后进入 Builder 页面。

Builder 页面分为两部分：

```txt
左侧：Agent Steps
右侧：App Preview
```

左侧展示 AI Agent 的执行步骤，例如：

- Understanding request
- Planning app structure
- Generating components
- Rendering preview

右侧根据 AI 返回的 `AppSpec` 渲染应用预览。

这部分的目标是让用户感知到：系统不是直接返回一段文本，而是在模拟一个智能体完成需求理解、产品规划、组件生成和预览渲染的过程。

---

### 3.4 SiliconFlow 大模型生成 AppSpec

项目通过服务端 API 调用 **SiliconFlow API**。

前端不会直接调用大模型，也不会暴露 API Key。

数据流如下：

```txt
前端 Builder 页面
  ↓
POST /api/generate
  ↓
Next.js API Route
  ↓
SiliconFlow Chat Completions
  ↓
返回结构化 AppSpec
  ↓
前端渲染 App Preview
```

大模型返回的不是 React 代码，而是一个结构化 JSON：

```ts
type AppSpec = {
  name: string;
  description: string;
  pages: string[];
  components: string[];
  features: string[];
  theme: "light" | "dark" | "colorful";
  interactivePreview?: InteractivePreview;
};
```

这样做的好处是：

1. 更稳定；
2. 更安全；
3. 不需要执行模型生成的代码；
4. 前端可以统一解释渲染；
5. 方便做 fallback；
6. 方便后续扩展成真实代码生成。

---

### 3.5 可交互 App Preview

为了避免预览只是静态展示，我将 AppSpec 扩展为包含 `interactivePreview` 字段。

例如对于个人记账应用，预览可以支持：

- 首页 / 记账页 / 统计页 / 账单明细页 tab 切换；
- 输入账单名称；
- 输入金额；
- 选择收入 / 支出；
- 选择分类；
- 添加记录；
- 删除记录；
- 动态更新收入、支出、余额统计。

对于其他类型应用，例如健身打卡、任务看板、学习计划工具，也会生成对应的交互表单、记录列表和统计卡片。

这个设计让 Demo 不只是“生成一个方案”，而是能让用户实际体验一个小型应用原型。

---

### 3.6 数据持久化

项目使用 **Neon PostgreSQL** 做数据持久化。

主要保存：

- 项目 ID；
- 用户 ID；
- 项目标题；
- 原始 Prompt；
- 生成的 AppSpec；
- 创建时间；
- 更新时间。

核心表结构：

```sql
CREATE TABLE IF NOT EXISTS projects (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id TEXT NOT NULL,
  title TEXT NOT NULL,
  prompt TEXT NOT NULL,
  app_spec JSONB NOT NULL,
  created_at TIMESTAMP DEFAULT NOW(),
  updated_at TIMESTAMP DEFAULT NOW()
);
```

所有查询都会带上 `user_id` 条件，保证用户隔离：

```sql
SELECT *
FROM projects
WHERE user_id = $1
ORDER BY created_at DESC;
```

项目支持：

- 保存项目；
- 查看项目列表；
- 查看项目详情；
- 删除项目；
- 更新项目 AppSpec。

---

### 3.7 历史项目与项目详情

用户可以进入 `/projects` 查看自己的历史项目。

项目卡片展示：

- 项目名称；
- 原始 Prompt 摘要；
- 创建时间；
- 主题；
- 页面数量；
- 功能数量；
- 查看按钮；
- 删除按钮。

点击某个项目后进入 `/projects/[id]`。

详情页展示：

- 项目名称；
- 原始需求；
- 创建时间；
- 更新时间；
- pages；
- features；
- components；
- 可交互 App Preview。

历史项目中的 App Preview 仍然可以交互，保证保存下来的不仅是静态文本，而是一个可继续体验的应用原型。

---

### 3.8 AI 继续迭代能力

在项目详情页，用户可以继续输入修改指令，例如：

```txt
改成暗色风格
```

```txt
增加一个月度统计页面
```

```txt
增加预算提醒功能
```

系统会将：

- 原始 Prompt；
- 当前 AppSpec；
- 新的修改指令；

一起发送给 `/api/iterate`，再由 SiliconFlow 生成新版 AppSpec。

用户可以先预览新版结果，再点击 `Save Changes` 手动保存。

交互流程如下：

```txt
输入修改指令
  ↓
调用 /api/iterate
  ↓
生成新版 AppSpec
  ↓
更新 App Preview
  ↓
显示 Unsaved changes
  ↓
用户点击 Save Changes
  ↓
PUT /api/projects/[id]
  ↓
更新 Neon 数据
```

这个功能让 miniAtoms 从一次性生成器变成了一个持续迭代的 AI 应用构建工具。

## 4. 技术选型与依据

### 4.1 Next.js

选择 Next.js 的原因：

- 支持 App Router；
- 可以同时写前端页面和后端 API；
- 适合隐藏 SiliconFlow API Key；
- 适合连接 Neon；
- 与 Vercel 部署天然适配；
- 更接近真实全栈产品结构。

由于项目需要登录、数据库和服务端 API，Next.js 比纯前端框架更适合。

---

### 4.2 TypeScript

TypeScript 用于保障 AppSpec、Project、PreviewRecord 等核心数据结构稳定。

本项目大量依赖结构化 JSON，如果没有类型约束，前端渲染和数据库保存都容易出错。

TypeScript 帮助我在开发阶段明确：

- 大模型应该返回什么结构；
- 前端应该如何渲染；
- 数据库返回如何映射；
- fallback 如何兜底。

---

### 4.3 Tailwind CSS

Tailwind 用于快速实现高质感 UI。

项目需要还原类似 极简清透的视觉风格：

- 大渐变背景；
- 毛玻璃卡片；
- 大圆角输入框；
- 轻量阴影；
- 响应式布局。

Tailwind 能够快速完成这些样式，同时减少自定义 CSS 成本。

---

### 4.4 Clerk

Clerk 负责注册、登录、用户状态和路由保护。

选择 Clerk 的原因：

- 快速接入；
- 提供登录 / 注册弹窗；
- 提供 UserButton；
- 支持服务端鉴权；
- 适合 Next.js；
- 能快速满足题目中“注册 / 核心流程”的要求。

---

### 4.5 SiliconFlow API

SiliconFlow 用于实际大模型生成能力。

项目没有使用纯 Mock，而是接入真实大模型 API，让用户输入不同需求时可以生成不同应用方案。

同时我没有让大模型直接生成可执行代码，而是生成结构化 AppSpec。这样在 6–8 小时开发周期内更稳定，也更容易保证 Demo 可运行。

使用硅基流动平台的API还考虑其包含大部分国产可用模型的调度实现，可用为后续项目的Multi-Agent模型选型提供统一接入。

---

### 4.6 Neon PostgreSQL

Neon 用于项目数据持久化。

选择 Neon 的原因：

- Serverless Postgres；
- 适合部署到 Vercel；
- 可以保存 JSONB；
- 配置简单；
- 足够支撑本次 Demo；
- 后续可以扩展为更完整的数据模型。

---

### 4.7 Vercel

Vercel 用于在线部署。

选择 Vercel 的原因：

- 对 Next.js 支持好；
- GitHub 推送后自动部署；
- 环境变量配置方便；
- 适合快速交付可测试在线链接。

---

## 5. 关键工程取舍

### 5.1 不让模型直接生成 React 代码

虽然题目是“通过智能体驱动完成代码或应用生成”，但在有限时间内，如果直接让模型返回 React 代码并执行，会带来很多问题：

- 输出不稳定；
- 代码可能无法编译；
- 存在安全风险；
- 需要沙箱运行环境；
- 调试成本高；
- 很容易影响最终可交付性。

因此我选择：

```txt
模型生成结构化 AppSpec
前端解释 AppSpec 并渲染交互式应用预览
```

这是一种更稳健的 Agentic App Builder 实现方式。

后续如果继续扩展，可以再把 AppSpec 转换为真实 React 文件或 Sandpack 代码。

---

### 5.2 使用 Neon 而不是 localStorage

为了满足真实数据持久化和用户隔离，我选择 Neon PostgreSQL，而不是 localStorage。

localStorage 开发更快，但只能做单机保存，不利于在线 Demo 的完整体验。

Neon 可以让用户：

- 换浏览器仍可看到项目；
- 登录后查看自己的历史项目；
- 保存和删除项目；
- 后续支持版本历史。

---

### 5.3 使用 Clerk 而不是直接代码实现 Auth

注册登录是主要功能但是直接代码实现会占用大量测试时间，包括：

- 密码加密；
- Cookie / Session；
- 表结构；
- 登录态维护；
- 保护路由；
- 用户头像；
- 登出。

本项目的重点是 AI Agent 产品体验，不是 Auth 系统本身。因此使用 Clerk 可以把时间集中在核心生成与交互体验上。

---

### 5.4 不做复杂版本管理

当前继续迭代功能采用：

```txt
生成新版 AppSpec → 用户确认 → 覆盖当前项目
```

没有做多版本历史。

原因是版本管理会引入额外表结构和 UI 复杂度。本次 Demo 先保证核心闭环，后续再扩展版本历史与回滚。

- 交互真实可用；
- 数据可持久化；
- 工程结构清晰；

- Demo 可部署、可测试、可扩展。
